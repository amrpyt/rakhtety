# Phase 06.1 Context

## Goal

Close the blockers found in the v1.0 milestone audit so the milestone can be archived with proof.

## Locked Decisions

- Keep this as one urgent closure phase after Phase 6.
- Do not bring back the removed Phase 7/8 roadmap ideas.
- Fix the audit blockers in the current Next.js/Supabase app.
- Use server API routes for permission-sensitive mutations.
- Use Browser Use for live browser checks when a browser check is needed.

## Audit Blockers

1. Client intake is not atomic enough.
   - Required documents must be sent with client creation.
   - If document upload fails, the client must not remain saved as valid.

2. Dashboard summary API uses browser-wired repositories.
   - Server route must use the request-bound Supabase server client.

3. Employee edit/delete flows use browser-side service paths.
   - Update, deactivate, reactivate, and delete must use API routes with `manageEmployees`.

4. Phase proof is incomplete.
   - Phase 1 and Phase 2 need verification records.
   - Phase 6 needs a clear final deployment decision.

## Success Criteria

- Client creation with intake files is one server request.
- Dashboard summary route uses the authenticated server Supabase client.
- Employee update/delete/deactivate/reactivate are server-gated.
- Phase 1 and Phase 2 verification documents exist.
- v1.0 milestone audit can be rerun without the same blockers.
