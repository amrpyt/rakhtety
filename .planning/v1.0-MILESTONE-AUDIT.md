## Cross-Phase Issues

1. **Client intake is not atomic**
   - Affected: CRM-01, CRM-04
   - Evidence: `src/lib/client-data/directory-client.ts`, `src/app/api/clients/route.ts`
   - Plain English: the app saves the client first, then uploads files. If the files fail, the bad client is still saved.

2. **Dashboard API uses the wrong Supabase path**
   - Affected: DASH-01..05, FIN-05, EMP-03
   - Evidence: `src/app/api/dashboard/summary/route.ts`, `src/lib/services/dashboard-summary.service.ts`, `src/lib/database/repositories/workflow.repository.ts`
   - Plain English: the server route asks the database like a browser, not like a logged-in server request.

3. **Employee edit/delete is not fully server-gated**
   - Affected: EMP-01, AUTH-04
   - Evidence: `src/hooks/useEmployees.ts`, `src/lib/services/employee.service.ts`
   - Plain English: create employee uses the safer road, but edit/delete still take the older road.

4. **Payment recording permission gate is inconsistent**
   - Affected: FIN-03
   - Evidence: `src/hooks/useFinancials.ts`, `src/lib/services/financial.service.ts`, `src/lib/database/repositories/financial.repository.ts`
   - Plain English: RLS may protect it, but app-level permission checks are not as clear as other flows.

## Broken Flows

| Flow | Break Point | Fix Direction |
|------|-------------|---------------|
| Add client with mandatory attachments | Client saved before files are safely uploaded | Make create+documents one server flow, or rollback client if upload fails. |
| Dashboard summary load | API route uses browser Supabase repositories | Pass request-bound server Supabase client into dashboard data queries. |
| Employee edit/delete | Browser service path skips API permission pattern | Add `PATCH`/`DELETE /api/employees/[id]` with `manageEmployees` permission. |
